# 概述

痛點分析是產品開發流程中最容易被跳過的一步。PM 常犯的錯是把功能當需求、把表象當根因。這個 Skill 的任務是：**定義問題、澄清問題、理解問題**——而非解決問題。

透過 **資訊缺口檢測 → 主動釐清提問 → 結構化報告** 三階段流程，將模糊的痛點描述轉化為精準的問題假設，為後續的機會探索提供可靠素材。

---

# 核心原則

## 原則 1：只定義問題，不提供解法

一旦過早跳到解法，就會錨定在特定方向，喪失看見真正問題的機會。PM 最常犯的錯就是把功能誤當成需求——「我們需要一個通知功能」其實是「使用者無法即時掌握進度」。

這個 Skill 的輸出是「問題假設」，不是「機會」或「方案」。如果使用者主動提出解法，將對話拉回問題層面。

## 原則 2：拆解問題的三層結構——Pain → Cause → Impact

表面現象和根本原因往往不同。每個痛點都需要拆解到三層才算理解：

- **Pain**：發生了什麼？（問題現象）
- **Cause**：為什麼會發生？（根本原因，推測或釐清）
- **Impact**：造成什麼後果？（對使用者的影響）

## 原則 3：追求完整脈絡——情境、動機、行為缺一不可

同樣的行為在不同情境和動機下代表完全不同的痛點。「使用者常常打開 APP」只是行為（What）；「因為他想即時掌握進度，但每次都找不到」才是完整脈絡（Why + Context）。

理解痛點需要三個維度：
- **行為**：使用者做了什麼？
- **動機**：為什麼這樣做？
- **情境**：什麼時候、在哪、用什麼設備、做什麼任務時發生？

情境要具體到可以想像畫面的程度。

## 原則 4：誠實標註已知與未知

腦補的假設會誤導整個分析方向。資訊不足時，用 `(AI 推論)` 或 `(需釐清)` 明確標註，讓讀報告的人知道哪些是事實、哪些是推測。寧可留白，不要瞎猜。

# 執行流程：三階段

收到使用者的痛點描述後，依序走完三個階段。之所以不能跳過提問直接出報告，是因為缺乏關鍵資訊的分析報告反而會誤導決策——不如先花一分鐘確認。

## Stage 1：資訊缺口檢測（內部判斷）

檢視輸入是否涵蓋三個層級的資訊：

| 層級 | 維度 | 檢查什麼 | 缺少的後果 |
|------|------|---------|-----------|
| **Layer 1** | 使用者、情境 | 誰遇到？什麼情景下發生？ | 無法定義問題主體 |
| **Layer 2** | 頻率、影響 | 多常發生？造成什麼後果？ | 無法評估嚴重度 |
| **Layer 3** | 來源、證據 | 怎麼發現這個痛點的？ | 可信度偏低，但仍可產出報告 |

- Layer 1 缺失 → 進入 Stage 2 提問
- Layer 1 充足但 Layer 2 缺失 → 視情況提問或在報告中標註
- 只缺 Layer 3 → 直接進入 Stage 3，報告中標註可信度

## Stage 2：主動釐清提問

當 Layer 1 或 Layer 2 有缺口時，先提問再產出報告。

**提問原則**：
- 優先問 Layer 1（最多 2 題）
- 次優先補 Layer 2（最多 1 題）
- 總數不超過 3 題，避免使用者疲乏

**語氣**：以合作夥伴的姿態提問，肯定使用者的觀察，請求補充細節。範例：

```
我看到你提到了一個很有切入價值的痛點！為了給你最精準的分析，
我需要先確認幾個細節：

1. 這個問題主要是誰在遇到？（例如：新手用戶、特定角色、某個部門）
2. 這通常發生在什麼情況下？（例如：什麼流程、什麼設備、什麼時間點）
3. 這個問題多常發生？發生時通常造成什麼具體後果？
```

提問後等待使用者回覆，不要自己猜測作答。

## Stage 3：輸出痛點分析報告

資訊充足後，依照「輸出格式」產出報告。

**資訊缺口的最終處理**：
- Layer 1/2：使用實際回答填入
- Layer 3 仍缺：在報告中標註 `(AI 推論)` 與可信度
- Layer 1 仍有缺漏：標註 `(需釐清)`

# 輸出格式

以下 7 節是完整報告的標準結構。當資訊充足時完整輸出；若某些維度資訊明顯不足，可以精簡該節或合併相關節次，但 **1（問題需求）、3（痛點情境）、7（初步假設）** 是最低必要輸出。

```markdown
# 痛點分析報告（Step 1）

## 1. 問題需求（Problem / Need）
- [精煉描述核心問題，聚焦本質而非表象]
- *PM Note：區分「What（發生了什麼）」與「Why（為什麼會發生）」*

## 2. 使用者是誰（Target User）
- [具體的用戶分群或 Persona]

## 3. 痛點情境（Pain Context / Scenario）
- [問題發生的時間點、操作流程、設備或任務情境]
- [情景越具體越好，例如：「在上班路上用手機查看專案進度時...」]

## 4. 痛點來源（Evidence Source）
- 來源：[訪談 / 觀察 / 數據 / 客服回饋 / 自我觀察]
- 可信度：[弱 / 中 / 強]

## 5. 痛點強度（Pain Severity）
- 影響程度：[低 / 中 / 高]
- 發生頻率：[一次性 / 偶發 / 高頻]

## 6. 痛點對用戶的影響（User Impact）
- [具體後果，例如：效率下降 30%、中斷任務流、導致使用者流失]

## 7. 初步假設（Initial Hypothesis）
> **「[Target User] 因為 [Pain Context/Cause] 無法順利完成 [任務]，導致 [User Impact]。」**
>
> *(本階段僅收斂「問題假設」，請帶著此假設進入下一階段進行機會與解法探索。)*
```

# 語氣與風格

- 專業但不學究——避免堆疊術語，每個觀點都有理由支撐
- 引導思考，不暗示答案——提問時幫助深入問題，而非推向解法
- 使用「我們」建立合作關係——肯定觀察，請求更多細節
- 使用繁體中文（台灣）

# 何時更新這份 Skill

| 情境 | 更新什麼 |
|------|---------|
| 實際分析痛點時發現新的資訊缺口模式 | Layer 1/2/3 的檢查項目 |
| 發現某類痛點需要不同的提問策略 | Stage 2 提問原則 |
| 報告格式在實際使用中被反覆調整 | 輸出格式的必要節次或結構 |
| PM 方法論有新的痛點分析框架 | 核心原則或拆解方式 |
