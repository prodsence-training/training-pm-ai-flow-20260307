## 1. 問題需求（Problem / Need）

- **What**：敏捷團隊在 Sprint 執行過程中，各角色對「現在整體進度到哪」「哪裡卡住了」「插單佔了多少資源」缺乏共同的即時可見資訊，導致問題在 Sprint 後半段才爆發
- **Why**：現有工具（Jira）的進度資訊分散在多層報表中，沒有一個統一的、低摩擦的介面讓不同角色在各自的工作節奏中即時感知全局狀態。每個人只看到自己的 ticket，缺乏「我的工作 ↔ 產品大方向」的連結

*PM Note：5 位受訪者描述的表面現象不同（看不到進度、插單失控、站會流於形式、Roadmap 停滯），但根因指向同一件事——Sprint 過程中的資訊不透明與不同步。*

## 2. 使用者是誰（Target User）

這個痛點橫跨整個敏捷團隊，但痛感強度和角度因角色而異：

| 角色 | 痛感角度 | 代表受訪者 |
|------|---------|-----------|
| **工程師**（初/中階） | 不知道自己的工作跟大方向的關係；來不及發現自己落後 | Andy、Bob |
| **PM / PO** | 團隊沒有共同畫面，Roadmap 進展被隱性侵蝕 | Cindy |
| **Scrum Master / Tech Lead** | 站會無法暴露真實風險，插單缺乏判斷依據 | Darcy |
| **管理層** | 看不到團隊真實狀態，只能聽到「快好了」 | Emily |

## 3. 痛點情境（Pain Context / Scenario）

痛點集中在 **Sprint 執行期間**的三個關鍵時刻：

1. **每日站會時**：大家報告「昨天做了什麼、今天要做什麼」，但沒有人提阻礙，也沒有視覺化資訊幫助識別哪些 ticket 停滯、哪些插單正在吃掉原本的 Sprint capacity（Darcy）
2. **Sprint 中後段（Day 6-8）**：工程師才驚覺進度落後，此時已來不及調整，只能加班趕工（Bob）；Scrum Master 才發現 API 卡住或前後端不一致（Darcy）
3. **Sprint Review / 季度回顧時**：PM 發現 Roadmap 上的大功能幾乎沒有推進，但每個 Sprint 的完成率數字看起來都還行——因為完成的大多是插單或非 Roadmap 項目（Cindy、Emily）

## 4. 痛點來源（Evidence Source）

- 來源：使用者訪談（5 位，涵蓋工程師、PM、Scrum Master、老闆四種角色）
- 可信度：**強**——多角色交叉驗證，且受訪者自發描述相似的問題模式

## 5. 痛點強度（Pain Severity）

- 影響程度：**高**——影響整個團隊的交付節奏、Roadmap 進展、以及管理層的決策品質
- 發生頻率：**高頻**——每個 Sprint（通常 2 週）循環發生，且每日站會都是潛在的錯失偵測機會

## 6. 痛點對用戶的影響（User Impact）

- **工程師**：Sprint 後半段被迫加班；長期缺乏成就感（不知道自己的工作意義）
- **PM**：Roadmap 交付持續延遲，需要反覆向團隊重新解釋願景與策略
- **Scrum Master**：風險延遲暴露（Day 8 才發現 vs Day 3 就能發現），站會失去實質功能
- **老闆**：無法基於真實資訊做資源調配決策，只能依賴 PM 的主觀彙報

## 7. 初步假設（Initial Hypothesis）

> **「敏捷團隊的各角色（工程師、PM、Scrum Master、管理層）因為 Sprint 執行過程中缺乏即時、統一的進度與風險可視化資訊，無法在正確的時間點（每日站會、Sprint 中段）識別風險並做出調整，導致問題在 Sprint 後半段集中爆發——工程師加班、Roadmap 停滯、管理層決策盲區。」**
