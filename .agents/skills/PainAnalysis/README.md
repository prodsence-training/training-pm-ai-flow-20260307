# PM 痛點分析（Step 1）

將模糊的痛點描述轉化為精準的問題假設。

## 用途

分析產品痛點、建立問題假設。專注於問題定義與假設建立，絕不跳到解法。

## 觸發時機

- 使用者提出產品痛點、使用者抱怨、訪談筆記
- 任何「使用者遇到什麼問題」的描述
- 即使輸入模糊（「用戶好像不太滿意」）也應觸發

## 核心特性

- 三階段流程：資訊缺口檢測 → 主動釐清提問 → 結構化報告
- 層級化資訊檢查（Layer 1/2/3）
- Pain → Cause → Impact 拆解
- 輸出 7 節結構化報告（最低 3 節必要）
