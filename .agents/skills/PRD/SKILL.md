# 概述

PRD（Product Requirements Document）是產品探索流程的收斂點。前面的 Step 1-4 是發散——發現痛點、探索機會、拆解故事、定義驗收條件。PRD 的任務是把這些探索成果**收斂為工程團隊可執行的規格文件**。

常見錯誤：
- PM 跳過探索直接寫 PRD → 需求不清、反覆修改
- AI 自動擴張範圍 → PRD 包含了沒討論過的功能
- 把 PRD 當創作 → 改寫了 User Story 和 AC 的原始意圖

這個 Skill 的核心立場：**PRD 是翻譯，不是創作。**

---

## 核心原則

### 1. 忠於來源

User Story 和 AC 是上游 Skill（Step 3-4）的產出。PRD 只能引用，不能改寫或新增。

- User Story → 原文引用，不改寫
- Acceptance Criteria → 原文引用，不新增
- PRD 的價值 = 補充產品規格（邊界、邏輯、流程），不是重新定義需求

### 2. 逐步澄清，確認後才生成

不一次問完所有問題。根據使用者提供的材料，智慧判斷哪些已知、哪些需要補問。

每個問題提供 3-5 個選項（A/B/C/D），降低回答門檻。

### 3. 推論必須透明

AI 基於上下文推導的內容，必須標示 *(AI 推論)*。使用者有權確認或否決任何推論。

### 4. 不擴張範圍

只處理使用者明確提出的功能。不主動建議「順便加上 XX」。Out-of-Scope 和 Scope 同等重要。

## 執行流程

### Phase 0：確認輸入模式

啟動時，先詢問使用者：

> 你有哪些現成的材料可以提供？
> - **A. 完整材料**：User Story + Acceptance Criteria（+ 痛點分析 / OST 更好）
> - **B. 部分材料**：只有其中一些（例如只有 User Story）
> - **C. 從零開始**：只有一個想法或需求描述

**Mode A/B**：解析材料 → 進入 Phase 1 智慧澄清
**Mode C**：引導使用者補齊材料：
1. 確認使用者目前有什麼（完全沒有？有模糊想法？）
2. 最低需求：至少需要 User Story + AC 才能開始寫 PRD
3. 引導使用者使用 Step 3（UserStory）和 Step 4（AcceptanceCriteria）
4. 如果連痛點都不清楚，建議從 Step 1（PainAnalysis）開始
5. 材料補齊後，重新進入 Phase 0

### Phase 1：智慧澄清

解析使用者提供的材料後，分兩層提問。

**智慧跳題判斷**：
- 材料中明確提到 → 帶出已知答案，請使用者一句話確認
- 材料中隱含但不確定 → 帶出推論並標示 *(AI 推論)*，請使用者確認
- 材料中完全沒提到 → 正式提問，提供選項

**必問（永遠確認，即使材料中有提及）**：

1. **功能目標（Feature Goal）**
   - 這次功能最主要的目標？（解決痛點 / 補強流程 / 新能力 / 提升效率）

2. **主要使用者（Primary User）**
   - 誰是主要使用者？（前台使用者 / 管理員 / 內部營運 / 多角色）

3. **Scope（包含什麼）**
   - 本次範圍涵蓋？（前端畫面 / 後端邏輯 / 資料庫 / 通知排程）

4. **Out-of-Scope（不做什麼）**
   - 哪些明確不處理？（UI 設計 / 後端重構 / 性能優化 / 資料移轉）

**按需（材料中沒提到才問）**：

5. **使用流程 Flow**
   - 使用者完成功能的大致流程？（單頁 / 多步驟 / Wizard / 嵌入現有流程）

6. **系統行為（System Behavior）**
   - 是否有自動化行為？（通知 / 狀態標記 / 排程 / API 連動）

7. **資料需求（Data Requirements）**
   - 是否涉及資料規則？（新欄位 / 新狀態 / 格式限制 / 欄位驗證）

**額外澄清**：

8. **成效指標（Metrics）**
   - AI 根據功能目標建議 2-3 個追蹤指標，請使用者確認或調整

### Phase 2：資訊整理與確認

所有澄清完成後，整理為結構化摘要：

- 功能目標
- 主要使用者
- Scope / Out-of-Scope
- 使用流程
- 系統行為
- 資料規則
- 已有的 User Story（原文）
- 已有的 AC（原文）
- 成效指標

> 「請確認以上整理是否正確？是否需要補充？」

**等待使用者確認後才進入 Phase 3。**

### Phase 3：產生 PRD Draft

依照 SPEC-XXX Feature Spec 模板結構輸出。

## 輸出格式：SPEC-XXX Feature Spec

```markdown
# SPEC-XXX-{功能名稱} - Feature Spec

## 功能概述

### 需求背景
（引用痛點分析 / OST / Story 來源）

### 功能描述
（一句話說明本功能）

### 預期影響
- 使用者影響
- 業務影響
- 技術影響

## 用戶故事 (User Story)
（原文引用，不改寫）

## Acceptance Criteria (驗收標準)
（原文引用，不新增）

## 產品規格

### 功能邊界
- 包含範圍
- 不包含範圍
- 影響範圍

### 業務邏輯
- 計算規則
- 判斷標準
- 狀態定義

### 業務邏輯流程圖（Mermaid）
（按需產出：使用者要求或流程複雜時才生成）
（不得發明不存在的分支，必要時標註 *AI 推論*）

### 相關文件
（API / 設計稿 / PRD 連結）

## 成效追蹤
- 核心追蹤指標
- 觀察指標

## 變更記錄
| 版本 | 日期 | 作者 | 變更內容 |
|------|------|------|----------|
| v0.1 | {today} | {author} | 初版草稿 |
```

## 與其他 Skills 的銜接

PRD 是產品探索工作流的 Step 5，接收 Step 1-4 的產出：

```
Step 1: PainAnalysis（痛點分析）     → 需求背景
Step 2: OST（機會解法探索）           → 功能目標、Scope
Step 3: UserStory（故事拆解）         → 用戶故事（原文引用）
Step 4: AcceptanceCriteria（驗收條件） → AC（原文引用）
Step 5: PRD（本 Skill）              → SPEC-XXX Feature Spec
```

**輸入越完整，澄清越少，PRD 品質越高。**

Mode C 的使用者如果缺少材料，應引導他們先使用對應的 Step Skill 補齊。
