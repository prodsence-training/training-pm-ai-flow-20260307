# 痛點分析報告（Step 1）

## 1. 問題需求（Problem / Need）
- **核心問題**：整個團隊在 Sprint 執行過程中，缺乏一個能將「日常任務與插單」與「高層級戰略目標（Epic/Roadmap）」相連結，並且能即時呈現進度與潛在風險的全局視角。
- *PM Note：表面上看似是「大家不知道進度」、「不斷有插單」或「站會流於形式」，但根本原因 (Cause) 在於現有工具（如 Jira）的資訊隱藏得太深，導致各角色只能看見自己眼前的單一任務（Silo）。這使得團隊無法評估插單對 Sprint 整體的衝擊，也無法及早暴露風險。*

## 2. 使用者是誰（Target User）
整個產品開發團隊，包含：
- **執行層**：後端工程師 (Andy)、前端工程師 (Bob)
- **管理與引導層**：PM/PO (Cindy)、Scrum Master / Tech Lead (Darcy)
- **決策層**：老闆 (Emily)

## 3. 痛點情境（Pain Context / Scenario）
這些痛點分散在產品開發流程的不同時間點與情境中：
- **日常開發（工程師）**：每天專注於眼前的 Ticket，遇到緊急插單就先做。往往要到 Sprint 第 6、7 天，才驚覺原本規劃的進度落後，必須開始加班。
- **每日站立會議（Scrum Master）**：站會時大家只做表面的進度報告（做完什麼、要做什麼），很少有人能主動意識到並提出阻礙（如 API 卡住、前後端不一致），通常要到第 8 天才發現風險。
- **Sprint 執行中（PM）**：雖然同意了插單，但看著 Sprint 推進，卻發現 Roadmap 上的中長期關鍵功能毫無進展。雖然常講願景，但團隊覺得跟自己每天做的事無關。
- **跨 Sprint / 季度檢視（老闆）**：詢問進度時總得到「快好了」的回應，但從複雜的 Jira 報表中無法直觀看出 Epic 實際推進的程度和最大風險在哪裡。

## 4. 痛點來源（Evidence Source）
- **來源**：使用者訪談（團隊內部 5 位不同角色的觀察與回饋）
- **可信度**：強（各角色的痛點雖然切入點不同，但相互印證，皆指向「進度盲區」與「目標脫節」）

## 5. 痛點強度（Pain Severity）
- **影響程度**：高（直接導致加班、Roadmap 延遲、團隊士氣與目標感低落）
- **發生頻率**：高頻（每天的站會都在發生，每個 Sprint 末期反覆上演加班與進度落後的戲碼）

## 6. 痛點對用戶的影響（User Impact）
- **對工程師**：經常被迫在 Sprint 到期前趕工加班，且淪為「解票機器」，對產品整體缺乏成就感與方向感。
- **對 PM / Scrum Master**：無法及時識別插單的影響與開發阻礙，導致敏捷站會失去「檢視與調適」的核心意義，最終 Roadmap 承諾屢屢跳票。
- **對管理決策層**：因為看不到真實的進度與風險，無法在正確的時機給予團隊資源協助或方向引導。

## 7. 初步假設（Initial Hypothesis）
> **「產品開發團隊」** 因為 **「在日常開發與站會的情境中，缺乏一個能直觀對齊『眼前任務、插單衝擊』與『宏觀目標進展、潛在風險』的資訊視角」** 無法順利完成 **「及早的風險預警與目標聚焦」**，導致 **「常常在 Sprint 最後一刻才發現進度落後而加班，且最終交付往往偏離了產品長期的 Roadmap。」**
