# Jira Dashboard MVP - 產品背景（Gemini 知識庫）

**用途**：Jira Dashboard（資料來源：Google Sheet）的產品邊界定義
**對象**：供 Gemini Gem（OST、User Story、AC 等）上傳使用

---

## 產品基本資訊

**產品名稱**：Jira Dashboard（敏捷團隊進度監控板）

**產品簡述**：展示 Jira 資料的 Web Dashboard，資料來源為 Google Sheet。幫助敏捷團隊實時掌握 Sprint 進度、Issue 分布和團隊容量狀態。

**核心用戶**：敏捷團隊中的工程師、PM、Scrum Master、管理層

---

## ✅ 你應該探索的改進方向

可以建議以下方向的解決方案和改進（在此範圍內發散）：

**UI/UX 層面**
- Dashboard 版面佈局和卡片配置優化
- 視覺階層、色彩編碼、資訊設計改進
- 互動體驗增強（篩選、排序、搜尋、鑽取）

**指標與可視化**
- 新增進度指標（如 Burndown Chart、Velocity 趨勢、Cycle Time）
- 風險預警機制（延遲預警、瓶頸檢測、異常警示）
- 多維度分析視圖（按優先度、Issue Type、Assignee、團隊等）

**資料與邏輯**
- 資料轉換與計算邏輯（如何從原始 Jira 資料計算有意義的指標）
- 資料篩選、分組、排序邏輯設計
- 與原始資料的可追蹤性（使用者能清楚看到數據來源和計算過程）

**功能擴展**
- 不同角色的檢視客製化（Engineer view vs PM view vs Manager view）
- 實時更新與同步機制（與 Google Sheet 的同步策略）
- 通知與警示機制（Sprint 即將結束、風險 Issue、變更影響）

**具體例子**：
- 建立「各角色專用的 Dashboard 檢視」，讓工程師看到自己的任務進度，PM 看到整體 Sprint 進展
- 設計「風險預警指標」，在 Sprint 中期就能識別延遲風險
- 建立「插單衝擊可視化」，量化變更對 Sprint 容量的侵蝕
- 設計「實際進度 vs 計畫進度」的對比展示
- 添加「團隊速度趨勢」分析，幫助改進估算精度

---

## ❌ 絕對禁止建議超出範圍的解法

以下方向應該完全避免（超出產品邊界）：

- ❌ 改變客戶的 Jira 工作流程（如：改 ticket 狀態、改 sprint 定義、改優先度規則）
- ❌ 改變客戶的組織流程（如：改 standup 時間、改 sprint 長度、改工作方式）
- ❌ 客戶系統層面的改動（如：Jira 外掛設定、第三方工具整合、API 架構修改）
- ❌ 超出 Dashboard 展示與資料加工層的技術決策
- ❌ 要求改變 Google Sheet 的資料結構（23 欄位固定 schema）

**為什麼禁止**：
- 這些決策超出我們的產品邊界
- 我們無法控制客戶的組織流程或 Jira 設定
- 我們的價值在於「可視化與洞察」而非「改變工作方式」

---

## 核心問題框架

無論在痛點分析、機會探索或需求定義中，始終聚焦於以下核心問題：

> 「**我們的 Dashboard 如何將 Google Sheet 中的 Jira 資料轉化為可理解、可操作的視覺化資訊，幫助敏捷團隊及時發現風險、掌握進度、做出更好的決策？**」

**不聚焦於**：「Jira 工作流程應該怎麼改」、「組織流程應該怎麼改」、「客戶的工作方式應該怎麼改」

**聚焦點**：可視化設計、資料呈現、指標計算、使用體驗

---

## 資料來源

本 Dashboard 的資料來自 **Google Sheets**（公開 CSV API，無需 API 金鑰）：
- **rawData 工作表**：包含 Jira Issue 的原始資料
- **GetJiraSprintValues 工作表**：提供 Sprint 選項

這是本 Demo Project 的基本約束，所有改進建議應以此為基礎。
