🧩 痛點分析報告（Step 1）v2.0
1. 問題需求（Problem / Need）
核心問題： 團隊雖然在 Jira 記錄了所有工作，但 缺乏對「插單影響（Unplanned Work）」與「任務滯留狀態（Flow Stagnation）」的量化可視性。

具體落差： 現有的 Burn-down Chart 過於粗略（僅顯示剩餘點數/時間），無法顯示 Ticket 具體卡在哪個環節（如：卡在 Code Review 還是 QA？），也無法區分「計畫內工作」與「插單工作」的比例與來源，導致團隊無法在 Sprint 期間識別真正的阻礙與風險。

2. 使用者是誰（Target User）
第一線受害者： 工程師（Andy, Bob）— 被迫處理插單但不知道這佔據了多少產能，且無法自發察覺進度瓶頸。

管理與協調者： Scrum Master (Darcy)、PM (Cindy) — 雖有數位工具，但仍需依賴口頭詢問才能得知「為什麼卡住」，無法進行數據驅動的風險控管。

盲區決策者： 老闆 (Emily) — 只能看到結果（延遲），無法看到過程中的資源被插單稀釋。

3. 痛點情境（Pain Context / Scenario）
情境： 團隊全權依賴 Jira 作為同步機制。

關鍵時刻：

每日站會時： 團隊看著看板或報表，但因為看不出「Ticket 在某狀態停留過久」，導致沒人提出阻礙。

Sprint 第 6-8 天： 累積的隱性延遲（卡狀態）與插單效應疊加，導致突然發現進度嚴重落後。

迭代回顧時： 因為沒有統計插單來源與比例，無法針對「干擾源」制定改善策略，導致問題重複發生。

4. 痛點來源（Evidence Source）
來源：訪談 + 用戶補充釐清（關於工具限制與數據盲區）

可信度：極強 (Strong)

PM Note：用戶明確指出工具功能限制（Burndown 太籠統）與流程缺失（沒統計插單），這是非常具體的歸因。

5. 痛點強度（Pain Severity）
影響程度：高

導致資源分配不明（不知道多少產能被插單吃掉）。

導致決策延遲（直到 Sprint 結束前才發現卡在特定狀態）。

發生頻率：持續性

只要使用 Jira 進行 Sprint，這個「數據黑箱」就一直存在。

6. 痛點對用戶的影響（User Impact）
認知失調： 團隊覺得自己一直在做事（Ticket 都有在動），但 Burn-down 沒有反應真實風險，直到最後一刻才崩盤。

無效優化： 由於不知道插單的「具體來源」與「佔比」，PM 與 Tech Lead 無法對症下藥（例如：若知是某業務單位常插單，可去溝通；若知是 Bug 多，可加強測試），導致只能被動接受干擾。

管理焦慮： 老闆與 PM 無法透過現有報表掌握現況，必須頻繁口頭確認，增加了團隊的溝通成本。

7. 初步假設（Initial Hypothesis）
我們從不同維度拆解出以下四個關鍵假設，這將成為我們下一步探索機會的基礎：

假設 A：數據盲區（Data Visibility）
團隊因為 [Jira 報表顆粒度不足（無法顯示狀態滯留）且缺乏對插單工作的分類統計機制]，導致 [無法在 Sprint 進行中識別「隱性瓶頸」與「產能被稀釋的程度」]，最終造成 [風險在 Sprint 尾聲才爆發，且無法針對干擾源進行源頭管理]。

假設 B：認知斷裂（Context Disconnect）
開發團隊因為 [缺乏在 Ticket 層級直接看見「業務價值」與「所屬 Epic 關聯」的機制]，導致 [將工作視為單純的清單執行，無法依據業務重要性自主判斷優先順序]，最終造成 [Roadmap 推進緩慢，且團隊對產品願景無感]。

假設 C：儀式失效（Ritual & Behavior）
團隊因為 [每日站立會議（Standup）流於形式化的流水帳報告，缺乏「針對阻礙（Blocker）」的提問文化與視覺引導]，導致 [即使有卡頓發生（如 API 不一致），成員也傾向隱忍或延後回報]，最終造成 [風險錯過最佳處理時機]。

假設 D：入口失控（Intake & Triage）
團隊因為 [缺乏明確的「插單過濾標準」與「專屬的緩衝通道」]，導致 [所有外部請求都被視為「緊急」並直接擠壓當前 Sprint 的產能]，最終造成 [Roadmap 的預定工作量被無形稀釋，且無法量化拒絕外部需求]。
